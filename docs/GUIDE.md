# OpenGauge Setup & Usage Guide

OpenGauge is a local-first PromptOps observability tool for LLM applications. It tracks per-call cost, token usage, runaway loop detection, and budget enforcement — all stored locally in SQLite.

## Quick Start

```bash
npx opengauge
```

This starts the local chat UI and opens it in your browser at `http://localhost:3000`. All data is stored locally at `~/.opengauge/`.

---

## Setup

### Standalone (Chat UI)

The simplest way to use OpenGauge — a local web chat interface with built-in token optimization and cost tracking.

```bash
npx opengauge
```

On first run, configure your provider either through the UI wizard or by creating a config file manually (see [Configuration](#configuration)).

To use a different port:

```bash
npx opengauge --port 3001
```

### OpenClaw Plugin

If you use [OpenClaw](https://openclaw.ai), OpenGauge can observe every LLM call your agent makes — without modifying your agent code.

**Install:**

```bash
openclaw plugins install @opengauge/openclaw-plugin
```

**Restart the gateway:**

```bash
openclaw gateway restart
```

That's it. The plugin hooks into OpenClaw's `llm_input` and `llm_output` events and logs every call automatically.

### Proxy Mode

OpenGauge can also run as a proxy between your application and your LLM provider, logging all requests that pass through.

```bash
npx opengauge proxy
```

## Configuration

All configuration lives in `~/.opengauge/config.yml`.

### Provider Setup

```yaml
providers:
  anthropic:
    api_key: YOUR_ANTHROPIC_API_KEY
    default_model: claude-sonnet-4-6

  openai:
    api_key: YOUR_OPENAI_API_KEY
    default_model: gpt-4o

  ollama:
    base_url: http://localhost:11434
    default_model: llama3

defaults:
  provider: anthropic
```

### OpenClaw Plugin Config

The OpenClaw plugin reads its settings from the `openclaw` section of `~/.opengauge/config.yml`:

```yaml
openclaw:
  circuit_breaker:
    enabled: true
    similarity_threshold: 0.8    # Jaccard trigram similarity (0-1)
    max_similar_calls: 5         # Trip after this many similar calls
    action: warn                 # warn | block

  budget:
    session_limit_usd: 5.00
    daily_limit_usd: 20.00
    monthly_limit_usd: 400.00

  optimize: false
  log_response_text: true        # Store assistant responses in DB
  log_full_request: false        # Store full message history (verbose)
```

If no config file exists, sensible defaults are used.

## Viewing Stats

### Stats Command

```bash
npx opengauge stats
```

This shows a full dashboard with:

- **Total Spend** — aggregate cost, tokens in/out, tokens saved
- **Spend by Source** — breakdown by source (chat, openclaw, proxy)
- **Model Distribution** — spend and token counts per provider/model
- **Daily Spend** — day-by-day cost and token totals
- **Top Sessions by Cost** — most expensive sessions
- **Active Alerts** — circuit breaker and budget warnings

### Filtering Options

```bash
# Filter by source
npx opengauge stats --source=openclaw
npx opengauge stats --source=proxy
npx opengauge stats --source=chat

# Filter by time period
npx opengauge stats --period=24h
npx opengauge stats --period=7d
npx opengauge stats --period=4w

# Filter by model
npx opengauge stats --model=claude-sonnet-4-6

# Show only active alerts
npx opengauge stats --alerts

# Combine filters
npx opengauge stats --source=openclaw --period=7d --model=claude-sonnet-4-6
```

### JSON Output

For scripting or piping into other tools:

```bash
npx opengauge stats --json
```

## What Stats Are Tracked

Every LLM call is logged with the following data:

| Metric | Description |
|--------|-------------|
| **Tokens In** | Input tokens sent to the model (includes prompt, system message, and cached tokens) |
| **Tokens Out** | Output tokens generated by the model |
| **Cost (USD)** | Estimated cost based on model pricing |
| **Latency** | Response time in milliseconds |
| **Provider** | Which LLM provider handled the call (anthropic, openai, gemini, ollama) |
| **Model** | The specific model used (e.g. claude-sonnet-4-6, gpt-4o) |
| **Source** | Where the call originated — `chat` (UI), `openclaw` (plugin), or `proxy` |
| **Session** | Calls are grouped into sessions (5-minute inactivity timeout creates a new session) |
| **Tokens Saved** | Tokens saved through prompt optimization and deduplication |

### Aggregated Views

The stats command rolls up individual calls into:

- **Total spend** across all time or a filtered period
- **Per-source breakdown** — see how much each integration costs
- **Per-model breakdown** — compare cost efficiency across models
- **Daily trends** — track spend over time
- **Session rankings** — find your most expensive sessions
- **Alert history** — runaway loops and budget breaches

## Safety Features

### Circuit Breaker

Detects when an agent is stuck in a loop making the same call repeatedly.

**How it works:** Uses trigram Jaccard similarity to compare recent prompts. If multiple consecutive calls exceed the similarity threshold, an alert is triggered.

| Setting | Default | Description |
|---------|---------|-------------|
| `enabled` | `true` | Turn circuit breaker on/off |
| `similarity_threshold` | `0.8` | How similar prompts must be to count as duplicates (0-1) |
| `max_similar_calls` | `5` | Number of similar calls before tripping |
| `action` | `warn` | `warn` logs an alert; `block` stops the call (OpenClaw plugin can only warn since hooks are observation-only) |

### Budget Enforcement

Set spend limits to prevent runaway costs:

| Setting | Default | Description |
|---------|---------|-------------|
| `session_limit_usd` | `$5.00` | Max spend per session |
| `daily_limit_usd` | `$20.00` | Max spend per day |
| `monthly_limit_usd` | `$400.00` | Max spend per month |

When a limit is breached, an alert is logged. In proxy mode with `action: block`, calls are actively blocked.

## Data Storage

All data is stored locally:

```
~/.opengauge/
  config.yml          # Configuration
  opengauge.db        # SQLite database (sessions, interactions, alerts)
  error.log           # Error log for debugging
  models/             # Local model files (if using Ollama)
```

The SQLite database contains:

- **sessions** — grouped LLM call sequences with aggregate stats
- **interactions** — individual LLM calls with prompt, response, tokens, cost, latency
- **alerts** — circuit breaker warnings and budget breach notifications

## Troubleshooting

### Plugin not loading in OpenClaw

Check the error log:

```bash
tail -50 ~/.opengauge/error.log
```

Common issues:
- **`better-sqlite3` bindings not found** — Run `cd ~/.openclaw/extensions/opengauge/node_modules/better-sqlite3 && npm run build-release`
- **`Could not get current provider`** — You're running an old version of the plugin. Update with `openclaw plugins install @opengauge/openclaw-plugin`

### Stats show $0.00 cost

- Model not in the pricing table — unknown models default to free. Check `npx opengauge stats` to see which model name is being reported.
- Tokens show 0 — the LLM provider may not be returning usage data in the expected format.

### sqlite-vec warning

```
sqlite-vec extension not found. Vector search will be unavailable.
```

This is harmless. Vector search is optional and used for RAG features in the chat UI. Core stats and cost tracking work without it.

### Resetting data

To start fresh, delete the database:

```bash
rm ~/.opengauge/opengauge.db
```

A new database is created automatically on the next run.
